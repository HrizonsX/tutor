## Why

当前插件端把许多 `selectionchange` 中间态也当成“用户选中了概念”处理，容易把逗号、半截词、拖拽未完成的文本写入记忆并触发解释。这个变更要把“选取完成”和“有效概念”作为前置门禁，让明确选中的概念继续被解释和记忆，噪声选择则安静忽略。

## What Changes

- 在内容脚本侧引入基于主按钮拖拽/释放的选取完成判断，并保留键盘选取、双击选词等非拖拽路径的稳定兜底。
- 在写入 `USER_SELECTED_TERM`、更新显式选取信号、触发解释评估前，统一校验选中文本是否是可解释概念。
- 拒绝标点/空白/符号-only、过大文本、代码块、明显半截英文词、过短且无上下文支撑的 CJK 片段等噪声选择。
- 为被拒绝的选择输出可诊断原因，便于从页面诊断字段和测试中确认“为什么没有触发解释”。
- 保持有效短概念、知识对象、技术缩写、中文词组、双击选词、键盘选取的现有可解释行为。

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `reading-context`: 选取行为必须区分“候选变化”和“用户完成一次选择”，只有完成且通过校验的选择才能成为当前阅读上下文的有效注意信号。
- `concept-understanding`: 显式选取概念必须通过噪声、边界、长度和上下文校验后才能被优先作为解释目标。
- `confusion-inference`: 标点、半截词、过大/代码选择等无效选择不得作为混淆或主动解释触发证据。
- `encountered-knowledge`: 用户选取事件写入知识记忆前必须经过同一概念有效性门禁，避免噪声知识对象污染记忆。

## Impact

- 主要影响浏览器插件内容侧：`src/content.js`、`src/behavior.js`、`src/concepts.js`、`src/reading-context.js`。
- 测试需覆盖概念校验、拖拽完成、键盘/双击选取、噪声选择不写记忆且不触发解释。
- 不引入新的后端、网关、模型 provider 或持久化依赖；外部 API 协议保持兼容。
