## Why

当前浏览器伴读插件已经能围绕技术阅读片段、行为信号和学习记忆生成低打扰解释，但用户在真实网页中遇到的理解障碍并不限于技术术语。历史典故、天文地理、科技圈名词、小说电影、组织人物、理论事件等泛知识对象，同样可能成为理解当前网页语义的关键。

本变更希望把系统从“技术概念伴读”扩展为“泛知识伴读”：由 agent 识别关键知识对象、检索用户记忆和偏好画像、决定讲不讲和讲什么，再由大模型生成短、准、贴上下文的解释，并让用户反馈持续影响未来行为。

## What Changes

- 新增泛知识对象识别与记忆能力：只要某个对象是理解当前网页语义的关键对象，就可进入候选范围，不限于技术术语。
- 新增 agentic 记忆整理与检索流程：agent 负责整理概念别名、历史解释、卡壳证据、相关对象和用户偏好，但必须保留不确定性和证据来源。
- 新增用户阅读画像：用户反馈会影响未来信号选择、解释优先级、知识类别偏好、解释风格和干预频率。
- 新增 LLM 短解释生成边界：agent 和策略层决定“讲不讲、讲什么”，大模型负责“怎么短、准、贴当前上下文地讲”。
- 新增解释卡片反馈入口：用户可以表达懂了、没懂、不准、别再提示、换个说法、需要更简单或更多背景等反馈。
- 新增重新生成解释按钮：重新生成不是随机刷新，而是携带上一版解释、用户反馈、偏好画像和目标改写风格生成新版本。
- 增强事实校验策略：稳定知识可由大模型直接保守解释；易变、近期、争议或事实敏感知识必须支持来源校验或降级处理。
- 保持现有安静伴读边界：agent 不点击、不跳转、不搜索页面、不操作浏览器内容，只在后台整理记忆、检索上下文和准备解释。

## Capabilities

### New Capabilities

- `encountered-knowledge`: 识别、归一化并维护用户在网页阅读中遇到的泛知识对象，包括人物、地点、组织、作品、历史事件、理论、文化典故、科技名词和其他语义关键对象。
- `user-reading-profile`: 根据用户反馈和阅读事件维护可解释、可清除的阅读偏好画像，用于影响未来信号选择、解释优先级、解释风格和静默策略。
- `short-explanation-composer`: 接入大模型生成短解释、改写解释和上下文角色说明，并约束其只负责表达，不负责最终干预决策。

### Modified Capabilities

- `concept-understanding`: 从技术概念提取扩展为泛知识对象识别，并在解释前接入 agentic 记忆检索。
- `learning-memory`: 从概念事件记忆扩展为知识对象事件记忆，并记录用户反馈、解释版本和 agent 生成的可追溯整理结果。
- `confusion-inference`: 在干预优先级中纳入用户阅读画像、类别偏好、显式反馈和重新生成行为。
- `cognitive-overlay`: 增加解释反馈控件、重新生成按钮和反馈事件记录，同时保持低打扰 UI。

## Impact

- 影响浏览器扩展内容脚本、overlay UI、记忆存储、概念/知识对象识别、干预策略、解释生成和测试用例。
- 需要新增 agentic 知识检索与用户画像相关数据契约，扩展现有 memory event 类型和 explanation prompt 输入输出结构。
- 需要接入 LLM 生成短解释与重新生成解释，并为事实敏感对象预留来源校验或保守降级路径。
- 需要新增隐私控制：最小化网页文本存储，优先保存知识对象、事件、偏好信号和少量上下文元数据，并允许用户清除或静默 learned preferences。
